# 1. 项目定位



## 项目名称



**ReAct Agent Workspace / 基于 ReAct 的智能工作台**



## 项目目标



做一个真正像样的 **Agent SaaS / 智能工作台**，让用户通过自然语言下发复杂任务，系统内部按 **ReAct 模式** 做循环执行：



- 先推理当前该做什么

- 决定要不要调用工具

- 通过 MCP 调用外部工具

- 根据工具返回结果继续推理

- 结合长期记忆调整输出

- 将长耗时任务交给后台异步执行

- 最后把结果、轨迹、记忆和任务状态都持久化



这类项目比普通“聊天机器人”强得多，因为它不是单轮问答，而是一个 **可推理、可执行、可扩展、可记忆、可追踪** 的 agent 系统。LangGraph 官方就把自己定位成构建、管理和部署 **long-running、stateful agents** 的底层编排框架；它同时提供了预构建的 `create_react_agent` 路线和专门的 ReAct template。([LangGraph](https://github.langchain.ac.cn/?utm_source=chatgpt.com))

# 3. 为什么这样选



## 3.1 LangGraph：最适合做 ReAct + Workflow



LangGraph 官方文档和模板都明确支持 ReAct agent。官方示例直接给出 `create_react_agent`，而 `langchain-ai/react-agent` 模板的 README 也明确写了：这是一个基于 LangGraph 的 ReAct agent 模板，核心逻辑会迭代地推理并执行 action，适合扩展到多工具场景。仓库星标约 **26.4k**，说明它已经不是边缘项目。([LangGraph](https://github.langchain.ac.cn/?utm_source=chatgpt.com))



所以你这里最合理的关系是：



- **ReAct**：决策范式

- **LangGraph**：运行时、状态编排、节点路由、执行轨迹



## 3.2 MCP TypeScript SDK：官方协议层



MCP 这里不要自己手搓协议，直接用官方 TypeScript SDK。官方 README 目前明确写着：**稳定的 v2 预计在 2026 年 Q1 发布，在此之前生产环境仍推荐 v1.x**。npm 侧也能看到它仍在持续发布，最近版本在 2026 年 2 月还在更新。这个信息非常关键：你项目可以追 MCP，但别一开始赌 v2。([GitHub](https://github.com/modelcontextprotocol/typescript-sdk?utm_source=chatgpt.com))



另外，官方 `modelcontextprotocol/servers` 仓库可以拿来参考实现，但它的 security 页面也明确写了：这里的 servers 是 **reference implementations / educational examples**，不是 production-ready；而且部分 server 已经迁到 archived 仓库以降低维护开销。所以你在项目里应把它当“样例和原型”，不是全盘照搬到生产。([GitHub](https://github.com/modelcontextprotocol/servers/security?utm_source=chatgpt.com))



## 3.3 Mem0：当前最容易写进简历的长期记忆层



Mem0 自己就把项目定位成 **Universal memory layer for AI Agents**，强调能记住用户偏好、适应个体需求并持续学习。公开资料也显示它到 2026 年 2 月还有 v1.0.x 系列 release，说明不是停更项目。([GitHub](https://github.com/mem0ai/mem0?utm_source=chatgpt.com))



## 3.4 Trigger.dev：适合长任务和可追踪执行



Trigger.dev 的定位非常贴你的需求：**long-running tasks、retries、queues、observability、elastic scaling**。这和 agent 系统里的文档解析、异步同步、批量处理、失败重试天然匹配。([GitHub](https://github.com/triggerdotdev/trigger.dev?utm_source=chatgpt.com))



## 3.5 Supabase：最省事的数据底座



Supabase 不只是数据库，它同时覆盖 Postgres、Storage、RPC、Realtime、向量检索等能力，GitHub 上约 **99k stars**。这让你不必一开始自己拆很多基础设施。([GitHub](https://github.com/supabase/supabase/contribute?utm_source=chatgpt.com))



## 3.6 Better Auth：新、活跃、适合 TypeScript 全栈



Better Auth 当前约 **27.2k stars**，而且 2026 年 3 月初还在快速发版，v1.5 系列明显很活跃。它还在新版本里增加了 **MCP authentication** 相关能力，这点和你的项目方向反而非常契合。([GitHub](https://github.com/better-auth/better-auth/stargazers?utm_source=chatgpt.com))

# 5. 核心执行流程



这是这套方案最关键的部分。



## 5.1 ReAct 主循环



用户输入：



> “分析我上传的项目和 GitHub 信息，给我生成一个适合写进简历的 agent 项目方案。”



系统内部流程：



1. **Receive Input**

   前端把用户消息发给 agent API。

2. **Load Session State**

   从 Supabase 读取当前会话上下文、历史任务、已连接工具。

3. **Retrieve Long-term Memory**

   从 Mem0 取用户偏好，比如目标岗位、常用技术栈、输出风格。

4. **Thought**

   ReAct agent 先生成当前阶段的推理：

   我需要先读取文档，再看 GitHub 信息，然后做结构化总结。

5. **Action Selection**

   LangGraph 决定调用哪些工具节点。

   例如：

   - 文件读取 MCP server

   - GitHub MCP server

   - 知识模板 MCP server

6. **Action Execution**

   通过 MCP client 调用工具。

7. **Observation**

   收到工具返回内容。

8. **Thought Again**

   根据 observation 再推理下一步要不要继续调用工具。

9. **Synthesis**

   生成结构化结果：

   - 项目名称

   - 技术选型

   - 模块拆分

   - 简历亮点

   - 下一步开发计划

10. **Write Back**

    将高价值结论存到：

    - Supabase：任务结果、轨迹、日志

    - Mem0：偏好、历史事实、稳定上下文

11. **Enqueue Async Jobs**

    如果有长耗时任务，比如文档分块、异步摘要、embedding、批量同步，则交给 Trigger.dev。



这个流程本质上就是：



**Thought → Action → Observation → Thought → Action → Observation → Final Answer**



而 LangGraph 正适合承载这种带状态的循环工作流。([GitHub](https://github.com/langchain-ai/react-agent?utm_source=chatgpt.com))

# 12. 最值得注意的坑



## 12.1 不要把 MCP servers 当成生产级直接拿来用



官方已经明确说它们主要是 reference implementations。你应该学习其实现方式，然后做自己的精简封装。([GitHub](https://github.com/modelcontextprotocol/servers/security?utm_source=chatgpt.com))



## 12.2 MCP SDK 先用 v1.x



官方现在仍建议生产环境优先使用 v1.x，别为了追新直接押 v2。([GitHub](https://github.com/modelcontextprotocol/typescript-sdk?utm_source=chatgpt.com))



## 12.3 记忆层不要等于“全量聊天记录”



Mem0 应该存 **高价值事实、偏好、稳定信息**，不是把所有上下文都无脑塞进去。



## 12.4 ReAct 需要上限控制



你要给 ReAct loop 设：



- 最大迭代次数

- 工具调用上限

- 超时

- fallback 策略



不然 agent 很容易反复打转。
